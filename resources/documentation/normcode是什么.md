# NormCode 是什么

本文档详细说明 NormCode 的定义、设计动机、核心概念、语法与格式、编译与执行，以及适用场景。

---

## 一、一句话定义

**NormCode** 是一种**半形式化语言**，用于编写**推理计划（Plans of Inferences）**——即把复杂任务拆成若干**数据隔离**的小步骤，供 AI 智能体按依赖顺序可靠执行。  
它像「数据库的 SQL」一样：学会 NormCode，就能用同一套语言描述任意 AI 智能体工作流，且不绑定具体框架、模型或平台。

---

## 二、NormCode 要解决什么问题

### 2.1 上下文污染（Context Pollution）

多步工作流中，若把多次大模型调用串在一起，常见问题包括：

- 步骤越多，上下文越长，模型容易**幻觉**、混淆中间结果、忘记任务约束。
- 难以精确定位「哪一步用到了哪份数据」——调试和审计成本高。
- 无法清晰区分「语义推理」（非确定性、耗 token）与「纯数据整理」（确定性、无 LLM 调用），成本和可靠性难以追踪。

**NormCode 的应对**：通过语言和运行时**强制数据隔离**——每一步只能看到你**显式传入**的输入，从结构上消除跨步污染。

### 2.2 可审计性与可调试性

在法律、医疗、金融等高 stakes 场景中，需要：

- 证明「每一步看到了什么、产出了什么」；
- 在出错时精确定位到某一步，并检查其输入输出；
- 支持断点、检查点与恢复。

NormCode 通过**显式数据流**和**唯一流程索引（Flow Index）**，让工作流在结构上可审计、可调试。

---

## 三、核心思想：推理与数据隔离

### 3.1 推理（Inference）作为基本单元

NormCode 围绕一个基本单元设计：**推理（inference）**。  
一个推理不是简单的「函数调用」，而是一条**结构化断言**：

> **概念 A** 是通过对 **输入 C、D** 执行 **操作 B** 得到的。

这种结构天然要求：  
- 每个步骤的输入、输出边界清晰；  
- 步骤之间没有隐式共享的「全局上下文」，只有显式传递的数据。

### 3.2 数据隔离（Data Isolation）

- 每一步是一个「密封房间」：只接收你在计划中写明的输入。
- 例如：摘要步骤**不能**看到原始长文档，只能看到上一步产出的「清洗后文本」。
- 带来的好处：
  - **减少幻觉**：模型不会把第 3 页和第 47 页的内容混在一起；
  - **降低成本**：不必在每一步都重新注入整份长文档；
  - **可审计**：可以逐步检查「该步的输入 / 输出」是什么。

## 四、基本语法：三个符号 + 自然语言

NormCode 在草稿格式（`.ncds`）中只需掌握三个结构标记，其余都可以是自然语言。

### 4.1 三个核心符号

| 符号 | 名称 | 含义 |
|------|------|------|
| `<-` | 值概念（Value Concept） | 「这是数据」——名词：文档、结果、列表等 |
| `<=` | 功能概念（Functional Concept） | 「这是动作」——动词：提取、判断、生成等 |
| `<*` | 上下文概念（Context Concept） | 循环/时序状态：迭代变量、条件分支中的 carried 数据等 |

### 4.2 阅读顺序：自底向上、缩进即依赖

计划按**缩进**表示依赖：子节点先于父节点完成。阅读时从最内层（叶子）往根读。

**示例**：

```ncds
<- document summary
    <= summarize this text
    <- clean text
        <= extract main content, removing headers and boilerplate
        <- raw document
```

含义（自底向上）：

1. 从 `raw document` 出发；
2. 执行 `extract main content...`，得到 `clean text`；
3. 再对 `clean text` 执行 `summarize this text`，得到 `document summary`。

**关键**：摘要步骤**看不到** `raw document`，只能看到 `clean text`。这就是数据隔离在语法中的体现。

### 4.3 黄金规则：一个推理只有一个动作

每个推理块中只能有**一个** `<=`（一个功能概念）。若需要多步，用嵌套表示：

```ncds
<- final output
    <= do step 2
    <- intermediate result
        <= do step 1
        <- input data
```

### 4.4 可选类型标注

编译器可根据自然语言推断类型；如需显式标注，可用 `|`：

- **值类型**：`object`（单条）、`relation`（列表/集合）、`proposition`（真/假）；
- **动作类型**：`imperative`（执行某操作）、`judgement`（判断真/假）。

## 五、概念类型：语义与句法

### 5.1 语义概念（会触发 Agent/LLM）

- **功能概念**（`<=`）带有一个 **norm**（范式/执行策略），决定由哪种 **Agent 序列**执行。
- **Imperative（祈使）**：执行命令、生成内容、调用工具 → 对应 **Imperative 序列**，通常涉及 LLM 或工具调用。
- **Judgement（判断）**：评估并返回真值 → 对应 **Judgement 序列**，通常也有 LLM 参与。

只有这类「功能概念」会触发**语义操作**（非确定性、消耗 token）。

### 5.2 句法操作（不调用 LLM）

以下操作是**确定性数据整理**，不调用 LLM，不消耗 token：

| 你写的自然语言（示例） | 编译器推断 | 在 .ncd 中的符号 | 是否 LLM |
|------------------------|------------|------------------|----------|
| 「把这些项收集在一起」 | 分组 | `&in` | 否 |
| 「从每一个里收集结果」 | 跨组收集 | `&across` | 否 |
| 「选第一个有效结果」 | 赋值/选择 | `$.` | 否 |
| 「若条件成立」 | 时序/条件 | `@if` | 否 |
| 「对列表中每一项」 | 循环 | `*every` 等 | 否 |
| 「分析…」「生成…」 | 祈使 | `::()` | 是 |
| 「判断是否…」「检查…」 | 判断 | `::<>` | 是 |

因此：一个计划可能有很多「步」，但只有一部分会真正调用 LLM；其余是分组、选择、条件、循环等句法操作。

### 5.3 Agent 序列（Sequences）

执行时，不同类型的功能概念会进入不同的**序列**，例如：

- **imperative**：执行复杂命令（常调用 LLM 或工具）；
- **judgement**：执行判断并得到布尔型结果；
- **grouping**：将多个引用组合成结构；
- **assigning**：赋值、选择、累积；
- **quantifying**：循环、迭代；
- **timing**：条件与顺序控制。

每个序列内部有固定的步骤（如 IWI、IR、QR 等），形成一条可复用的执行管线。

## 六、多格式生态（同一逻辑的不同视图）

NormCode 承认人、编译器、审核者需求不同，因此同一份「推理计划」存在多种**同构格式**：

| 格式 | 用途 |
|------|------|
| **.ncds**（NormCode Draft Straightforward） | 人类书写草稿：自然语言为主、结构标记最少，便于快速编写和阅读。 |
| **.ncd**（Draft） | 形式化中间表示：由编译器从 .ncds 生成，消除歧义（类型、绑定、流程索引等），供机器解析与执行。 |
| **.ncn**（Natural） | 自然语言叙述：由 .ncd 编译生成，把形式结构转成可读句子，便于领域专家验证逻辑。 |
| **.ncdn**（Hybrid） | 混合编辑：同一界面中并排显示 .ncd 与 .ncn，便于开发与审阅。 |
| **.nci.json** | 中间结构：表示概念与推理的树状关系。 |
| **concept_repo.json + inference_repo.json** | 执行期仓库：由 Activation 阶段生成，供 Orchestrator 加载并运行。 |

典型工作流：在 `.ncds` 中写计划 → 经编译得到 `.ncd`（及 `.ncn`）→ 再经后续阶段得到 `.nci.json` 和最终 JSON 仓库。

## 七、流程索引（Flow Index）与可寻址性

每个步骤在计划中都有一个唯一的**流程索引**（例如 `1`、`1.1`、`1.2.1`），由层级和兄弟顺序决定：

- 根节点为 `1`；
- 功能概念（`<=`）在父下通常为 `.1`；
- 同级的多个值/上下文概念依次为 `.2`、`.3`、`.4` 等。

作用：

- **精确定位**：报错或日志可指向「第 1.2.3 步」；
- **断点与干预**：可在指定步骤暂停、检查；
- **审计**：可查询任意一步的输入、输出；
- **跨引用**：步骤可显式依赖其他步骤的索引。

流程索引在形式化阶段生成，并贯穿整个执行与检查点恢复。

## 八、编译流水线（从草稿到可执行）

从人类可写的 `.ncds` 到 Orchestrator 可执行的 JSON，通常经历**四阶段**：

1. **Derivation（推导）**：可选，从自然语言或更高层描述推导出 .ncds。
2. **Formalization（形式化）**：`.ncds` → `.ncd`（及 `.ncn`）。  
   - 补全类型、绑定、流程索引，消除歧义。
3. **Post-Formalization（后形式化）**：对 `.ncd` 做进一步标注（如资源路径、需求声明），得到 `.pf.ncd`。
4. **Activation（激活）**：`.pf.ncd` → 解析为中间结构（如 `.nci`）→ 生成 `concept_repo.json` 与 `inference_repo.json`。  
   - 为每个推理生成 **working interpretation**，供各 Agent 序列使用；  
   - 校验并解析资源（demand/supply 模型）。

你主要维护 `.ncds`（或 `.ncdn`）；其余阶段由工具链自动完成。

## 九、执行：Orchestrator 与依赖驱动

### 9.1 Orchestrator 的角色

**Orchestrator** 是 NormCode 的运行时核心，负责：

- 按**依赖关系**调度推理（子节点完成后才执行父节点）；
- 维护**黑板（Blackboard）**：各概念与推理的状态（pending / in_progress / completed / skipped）；
- 与 **AgentFrames** 协作，按序列类型调用相应 Agent；
- 处理循环、条件、分支；
- **检查点（checkpoint）** 与恢复（常基于 SQLite 等持久化）。

计划定义「做什么」，Agent 定义「怎么做」，Orchestrator 定义「何时做」。

### 9.2 执行顺序

- **自上而下扫描**：按流程索引顺序（1 → 1.1 → 1.2 → 1.2.1 …）扫描等待列表。
- **自内而外完成**：只有当某推理的**所有子推理**都已完成，且其功能概念与值概念就绪时，该推理才会执行。
- 无依赖的多个推理可以并行执行（由调度器决定）。

### 9.3 检查点与恢复

Orchestrator 可将执行状态持久化。长时任务可在中断后从上一检查点恢复，而不必从头重跑。

## 十、半形式化：在表达力与严谨之间

NormCode 的「半形式化」体现在两方面：

1. **渐进形式化（Progressive Formalization）**  
   - 在保持结构（依赖、数据流）形式化的前提下，**内容**可以先用自然语言写（如 `::(summarize the text)`），之后再细化实现。  
   - 计划可以先当草稿迭代，再逐步收紧为严谨逻辑。

2. **仅在必要处严格（Strictness Only Where Necessary）**  
   - 语法仅在「编译所需」的维度上严格：概念唯一性、数据流依赖、working interpretation 的提取与序列选择。  
   - 在此之外，语义内容可以保留自然语言，避免像传统形式方法那样「一处语法错误整份规格失效」，同时仍保证编排可靠。

名字 **NormCode** 中的 "Norm" 指向**规范性推理**：智能体必须遵守的规则与范式。每一步都遵守「只看到显式给定内容」这一规范，且这种遵守在结构上可验证。

## 十一、何时使用 NormCode

| 场景 | 是否适合用 NormCode | 说明 |
|------|----------------------|------|
| 多步工作流（5 次以上 LLM 调用） | 适合 | 隔离与可调试性收益明显 |
| 需要可审计的 AI（法律、医疗、金融等） | 适合 | 每步输入输出可追溯 |
| 长时、可恢复的工作流 | 适合 | 内置检查点与恢复 |
| 快速原型（1～2 次 LLM 调用） | 通常不必 | 结构开销可能不划算 |
| 简单问答机器人 | 通常不必 | 直接调用模型即可 |

**总结**：NormCode 不负责让 AI 更聪明，而是让 AI 工作流**可检查、可隔离、可调试**——这对生产与高 stakes 场景往往更重要。

## 十二、与实现、论文和工具的关系

- **论文**：*NormCode: A Semi-Formal Language for Context-Isolated AI Planning*（如 [arXiv:2512.10563](https://arxiv.org/abs/2512.10563)）给出了语言设计、编译与执行的形式化描述。
- **实现**：本仓库包含编译器管线、Orchestrator、NormCode Deployment Server（FastAPI）、以及基于 React Flow 的 **Canvas App**（图可视化、断点调试、多 Agent 配置等）。
- **格式与示例**：仓库中可见 `.ncds`、`.ncd`、`.ncn` 示例，以及多位数加法、PPT 生成等完整用例。
